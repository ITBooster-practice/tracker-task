# CI для команды: быстрый старт

## Что такое CI простыми словами

CI (Continuous Integration) — это автоматическая проверка проекта на GitHub после каждого изменения.

Цель CI:

- поймать ошибки до попадания кода в основную ветку;
- держать `main` и `develop` в рабочем состоянии;
- сократить количество «сломанных» сборок после merge.

## Какие проблемы CI решает

- Код компилируется локально у одного разработчика, но не собирается у других.
- Тесты на локальной машине проходят, но в чистом окружении падают.
- В PR случайно попадают типовые ошибки линта/типов.
- Изменения в одной части монорепо ломают другую.

## Как CI работает в этом проекте

Конфиг находится в `.github/workflows/ci.yml`.

Пайплайн запускает отдельные проверки:

1. `lint` — правила качества и стиля кода.
2. `type-check` — проверка TypeScript типов.
3. `test` — тесты и порог покрытия.
4. `build` — проверка сборки проекта.

`build` запускается только если остальные проверки прошли успешно.

## Когда запускается

CI стартует на:

- `pull_request` в `main` и `develop`;
- `push` в `main` и `develop`.

Исключение:

- если изменена только документация (`docs/**`, `**/*.md`), CI не запускается.

## Почему это важно для PR

Если хотя бы одна проверка красная:

- PR нельзя считать готовым к слиянию;
- нужно исправить причину падения и дождаться зелёного CI.

## Что важно знать про тесты и покрытие

- Падение хотя бы одного теста делает job `test` красной.
- Порог покрытия — минимум 80%.

## Как читать статусы

- `green` — проверка пройдена.
- `red` — проверка упала, merge блокируется правилами ветки.
- `yellow/in progress` — проверка ещё выполняется.

## Что делать, если CI упал

1. Открыть упавший job (`lint`, `type-check`, `test` или `build`).
2. Прочитать первые ошибки в логе.
3. Повторить локально ту же команду.
4. Исправить проблему и запушить обновление в тот же PR.

## Минимальный локальный чек перед PR

1. `pnpm lint`
2. `pnpm type-check`
3. `pnpm test`
4. `pnpm build`

Эти шаги обычно совпадают с тем, что запускается в CI.
