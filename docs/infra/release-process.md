# Процесс релиза: практический гайд для команды

## Зачем нужен этот документ

Этот документ описывает рабочий процесс релиза в проекте: от фичи до выкладки, включая release-ветку, релиз-кандидаты (RC), smoke-проверки, теги, правила GitHub и базовые подходы к автоматизации.

Цель: чтобы любой участник команды понимал, что делать по шагам, без двусмысленности.

## Короткий словарь

- Release-ветка: ветка для стабилизации конкретной версии (например, `release/v1.0.0`).
- Freeze: режим, когда в release-ветку принимаются только блокеры и багфиксы, без новых фич.
- RC (Release Candidate): конкретный кандидат в релиз, фиксируется тегом вида `v1.0.0-rc.1`.
- Smoke-тесты: короткая ручная проверка критических пользовательских сценариев на собранном приложении.
- Финальный релиз: подтвержденный RC, который после merge в `main` получает тег `v1.0.0`.

## Базовый поток релиза (пошагово)

### 1. Разработка и интеграция

1. Фича делается в отдельной ветке от `main`.
2. Открывается PR в `main`.
3. CI проходит (линт, тесты, сборка, typecheck).
4. PR получает ревью и мержится в `main`.

### 2. Подготовка релиза

1. От актуального `main` создается `release/vX.Y.Z`.
2. В релизной ветке добавляется файл заметки релиза (например, `docs/releases/v1.0.0.md`).
3. Открывается PR `release/vX.Y.Z -> main`.
4. В PR объявляется freeze: только блокеры и багфиксы.
5. Ветка защищается правилами (PR required, без force push, без прямых update).

### 3. Стабилизация (release freeze)

1. Команда прогоняет релизный чеклист.
2. Найденные блокеры исправляются только в release-ветке.
3. Новые фичи в release-ветку не принимаются.

### 4. Релиз-кандидат (RC)

1. На текущий стабильный commit release-ветки ставится тег `vX.Y.Z-rc.N`.
2. Этот RC поднимается в тестовом окружении (или локальном staging-подобном окружении).
3. Проходится smoke-чеклист.
4. Если найден блокер: фикс в release-ветке, новый RC (`rc.2`, `rc.3`), повтор smoke.

### 5. Финал релиза

1. Когда RC подтвержден, release-ветка мержится в `main`.
2. После merge на `main` ставится финальный тег `vX.Y.Z`.
3. Публикуется GitHub Release с пользовательскими release notes.
4. Выполняется post-release smoke на целевом стенде.
5. Release-ветка удаляется, когда релиз подтвержден.

## Где и какой текст писать

### 1. Файл релиза в репозитории

Формат: `docs/releases/vX.Y.Z.md`.

Назначение:

- внутренняя фиксация состава релиза для команды;
- база для PR описания;
- база для будущих release notes.

### 2. PR release-ветки

Назначение:

- операционная коммуникация внутри команды;
- статус freeze;
- статус проверок;
- список ограничений и рисков.

### 3. GitHub Release

Назначение:

- человеческое описание для пользователей и стейкхолдеров;
- что реализовано, что исправлено, известные ограничения.

## Когда ставить теги

1. RC-теги (`vX.Y.Z-rc.N`) ставятся на release-ветке до финального merge.
2. Финальный тег (`vX.Y.Z`) ставится после merge в `main`, на commit в `main`, который реально выкатывается.

Почему так: в истории и в откате всегда будет однозначно видно, какой commit стал официальным релизом.

## Что такое smoke-тесты и чем они отличаются от CI

CI проверяет качество кода:

- линт,
- юнит/интеграционные тесты,
- сборка,
- typecheck.

Smoke проверяет критический пользовательский путь после сборки:

- вход в систему,
- открытие основных страниц,
- базовые критичные операции,
- критичные API.

Smoke обычно делают вручную (особенно на раннем этапе), потом постепенно автоматизируют самые важные сценарии.

## Минимальный smoke-чеклист

1. Авторизация: вход валидным пользователем.
2. Навигация: основные страницы открываются без ошибок.
3. Критичный CRUD-сценарий (например, проект или команда).
4. Критичный API возвращает ожидаемый ответ.
5. Выход из системы работает.
6. В логах нет критических ошибок после базовых действий.

## Стенд рядом с продом: как делают на практике

Да, релиз-кандидат можно поднимать на отдельном стенде рядом с продом.

Обычно:

1. Отдельный поддомен (например, `staging.example.com`).
2. Отдельные контейнеры и отдельные env.
3. Отдельная БД (никогда не продовая).
4. Отдельные ключи интеграций (sandbox).

## Ограничение доступа к стенду только для команды

Частые варианты:

1. IP allowlist.
2. Basic Auth в Nginx.
3. SSO/Cloudflare Access/VPN.
4. Комбинация VPN + Basic Auth.

Практичный старт для небольших команд: Basic Auth + HTTPS, позже перейти на SSO/VPN.

## Данные для staging: как не ломать прод

Нельзя использовать продовую БД напрямую для стенда.

Рабочий подход:

1. Периодический анонимизированный срез только нужных таблиц.
2. Лимит объема (последние N дней или N записей).
3. Доменные seed-данные для сценариев, где в срезе не хватает данных.
4. Обновление по расписанию (nightly или перед релизом).

### Что такое "пустые сущности"

Это когда в стенде нет данных, нужных для проверки сценария.

Примеры:

- есть пользователь, но нет команд;
- есть команда, но нет проектов;
- есть проекты, но нет задач.

Seed закрывает эту проблему, добавляя гарантированный минимальный набор данных для проверок.

## Проверка миграций и обратной совместимости

Даже если тесты зелёные, отдельно проверяется, что миграции безопасны для релиза:

1. Нет опасных изменений без переходного этапа (удаление/переименование колонок сразу).
2. Приложение продолжает работать в момент обновления.
3. Откат не ломает старую версию на новой схеме.

Практика: сначала расширяющие изменения схемы, удаление старого только в следующем релизе.

Для обязательных новых колонок это означает: сначала nullable + backfill, потом `NOT NULL`. Если данные уже есть в проде, нельзя рассчитывать на ручной seed как на способ миграции истории.

## Автоматизация: что внедрять поэтапно

### Этап 1 (минимум)

1. Автосборка и проверки по PR.
2. Ручной smoke-чеклист.
3. Ручной релизный тег.

### Этап 2

1. Автодеплой RC в staging по тегу `vX.Y.Z-rc.N`.
2. Полуавтоматический smoke (часть автотестами, часть руками).
3. Шаблон release notes.

### Этап 3

1. Автодеплой прод по тегу `vX.Y.Z`.
2. Автоматический rollback на предыдущий стабильный тег.
3. Полный аудит релизов и инцидентов.

## Роли и ответственность

- Тимлид: решение о freeze, подтверждение релизного состава, go/no-go.
- Разработчики: исправление блокеров, поддержка release-ветки.
- QA (если есть): smoke/регресс критичных сценариев.
- DevOps/платформа (если есть): деплой и контроль окружений.

## Чеклист тимлида перед финальным тегом

1. Release-ветка создана от актуального `main`.
2. Freeze объявлен и зафиксирован.
3. CI зеленый.
4. Smoke пройден.
5. Миграции проверены.
6. Известные ограничения зафиксированы.
7. RC подтвержден.
8. PR релизной ветки готов к merge.
9. После merge в `main` ставится финальный тег.
10. Публикуются release notes.

## Типичные ошибки

1. Пытаться создавать PR из release-ветки без diff к `main`.
2. Смешивать новые фичи и релизные фиксы после freeze.
3. Ставить финальный тег до merge в `main`.
4. Использовать продовые ключи/данные на staging.
5. Держать одновременно legacy branch protection и rulesets без понимания приоритета.

## Быстрый шаблон финального комментария в PR release-ветки

Релиз `vX.Y.Z`: собран стабилизированный срез.

Freeze:

- в release-ветку принимаем только blocker и bugfix;
- новые фичи в релиз не добавляем.

Статус:

- CI: зелёный;
- smoke: пройден;
- миграции: проверены;
- ограничения: зафиксированы в `docs/releases/vX.Y.Z.md`.

## Итог

Этот процесс можно запускать уже сейчас, даже без боевого сервера: локальный staging-подобный стенд в Docker + RC теги + ручной smoke. После появления VPS вы переносите те же шаги в реальное окружение без смены логики релизного потока.
