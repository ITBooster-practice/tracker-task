# Цветной вывод в таблице Coverage (v8)

За раскраску таблицы `% Coverage report from v8` отвечает **`watermarks`**, а не `thresholds`.

- **`thresholds`** — роняет тест-ран, если покрытие ниже порога
- **`watermarks`** — задаёт диапазоны для раскраски текстовой таблицы в терминале

## Где настраивать

Глобально для всех пакетов: `packages/vitest-config/src/base.ts`
Для конкретного приложения: `apps/web/vitest.config.ts` или `apps/api/vitest.config.ts`

## Пример конфига

```ts
coverage: {
  enabled: true,
  provider: 'v8',
  reportsDirectory: 'coverage',
  // Падает тест-ран, если покрытие ниже порога
  thresholds: {
    statements: 80,
    branches: 80,
    functions: 80,
    lines: 80,
  },
  // Отвечает за раскраску таблицы в терминале: [низкий, высокий]
  watermarks: {
    statements: [50, 80],
    branches: [50, 80],
    functions: [50, 80],
    lines: [50, 80],
  },
},
```

## Логика цветов (watermarks)

Формат `[низкий, высокий]`:

| Цвет    | Условие                      |
| ------- | ---------------------------- |
| Красный | значение < низкий            |
| Жёлтый  | низкий <= значение < высокий |
| Зелёный | значение >= высокий          |

## Актуальная стратегия исключений из coverage

Чтобы список `exclude` не разрастался из-за единичных файлов, в проекте используем правило:

- сначала исключения по шаблонам (категория файлов),
- точечные исключения — только если нет безопасного шаблона.

### Frontend (`apps/web/vitest.config.ts`)

Исключаются:

- тестовая инфраструктура: `test/**`,
- только презентационные UI-обёртки: `**/ui/layout.tsx`,
- скелетоны и state-заглушки: `**/ui/*-skeleton.tsx`, `**/ui/*-state.tsx`,
- orchestration-heavy page-view: `views/**/ui/*-page-view.tsx`.

Это снижает шум в branch-coverage и оставляет в метрике бизнес-логику.

### Backend (`apps/api/vitest.config.ts`)

Исключаются:

- thin wrappers и инфраструктура (`config`, `decorators`, `constants`),
- `guards/auth.guard.ts` как thin-wrapper над `AuthGuard('jwt')`,
- `mail/templates/**` как презентационный слой писем.

## Правило полезных тестов

- Пишем тесты для чистых функций и бизнес-веток (например, утилиты, мапперы, сервисная логика).
- Не добавляем хрупкие тесты только ради процентов покрытия на тяжёлые orchestration-компоненты.
- Если логика критична, выносим её из UI-обёртки в тестируемые функции/хуки и покрываем unit-тестами их.
