# Работа с Git

Проект использует методологию **Trunk-Based Development (TBD)**.

## Основные принципы TBD

- Работаем в `main` ветке (trunk)
- Краткосрочные feature-ветки (живут 1-2 дня максимум)
- Частые коммиты и мерджи
- Быстрая интеграция изменений

## Создание ветки

Формат названия ветки:

```
<тип>/<название-задачи>-<номер>
```

**Типы веток:**

- `feature` - новая функциональность
- `fix` - исправление бага
- `refactor` - рефакторинг кода
- `docs` - изменения в документации

**Примеры:**

```bash
git checkout -b feature/add-user-auth-42
git checkout -b fix/login-validation-15
git checkout -b refactor/api-structure-8
```

## Коммиты

### Формат сообщения

```
<тип>: <краткое_описание>
```

**Типы коммитов:**

- `feat` - новая функция
- `fix` - исправление бага
- `refactor` - рефакторинг
- `docs` - документация
- `test` - тесты
- `chore` - конфигурация, зависимости

**Примеры:**

```bash
git commit -m "feat: добавлен эндпоинт для создания задачи"
git commit -m "fix: исправлена валидация email"
git commit -m "refactor: упрощена логика аутентификации"
```

## Рабочий процесс

### 0. Взять задачу с доски

1. Открыть [доску проекта](https://github.com/orgs/ITBooster-practice/projects/1)
2. Выбрать задачу из столбца "Task"
3. Переместить в "In Progress - <твое имя>"
4. Назначить себя исполнителем (Assignees)

### 1. Создать ветку от main

```bash
git checkout main
git pull origin main
git checkout -b feature/my-task-123
```

### 2. Работать и коммитить часто

```bash
git add .
git commit -m "feat: добавлен базовый функционал"
```

### 3. Обновиться от main

```bash
git fetch origin
git rebase origin/main
```

### 4. Отправить изменения

```bash
git push origin feature/my-task-123
```

### 5. Создать Pull Request

- Описать изменения
- Запросить ревью
- Дождаться проверки CI/CD

### 6. Работа с доской задач

Доска проекта: https://github.com/orgs/ITBooster-practice/projects/1

После создания PR:

1. **Проверить задачу на доске** - убедись, что она в правильном столбце ("In Review")
2. **Скинуть ссылку в Telegram** - в соответствующую тему проекта
3. **Описать что сделано** - кратко напиши, что реализовано/исправлено
4. **Запросить approve** - попроси проверить

**Пример сообщения в Telegram:**

```
https://github.com/org/repo/pull/123

Реализовал аутентификацию пользователя:
- JWT токены
- Защита роутов
- Тесты покрытия
```

### 7. После мерджа удалить ветку

```bash
git checkout main
git pull origin main
git branch -d feature/my-task-123
```

Не забудь переместить задачу в столбец "Done" на доске! ✅

## Важные правила

- ⚠️ **Никогда не форс-пушить в main**: `git push --force` запрещён
- ✅ **Коммитить часто**: маленькие коммиты лучше больших
- 🔄 **Обновляться от main каждый день**: избегай больших конфликтов
<!-- - 🧪 **Тесты должны проходить**: перед пушем проверяй `pnpm test` -->
- 📝 **Понятные сообщения**: человек должен понимать, что изменилось

## Решение конфликтов

При конфликтах во время rebase:

```bash
# Посмотреть конфликты
git status

# Решить конфликты в редакторе
# Затем добавить файлы
git add <файл>

# Продолжить rebase
git rebase --continue
```

Если что-то пошло не так:

```bash
git rebase --abort
```

## Полезные команды

```bash
# Посмотреть статус
git status

# История коммитов
git log --oneline

# Посмотреть изменения
git diff

# Отменить локальные изменения
git checkout -- <файл>

# Вернуть последний коммит (не потеряв изменения)
git reset --soft HEAD~1
```
